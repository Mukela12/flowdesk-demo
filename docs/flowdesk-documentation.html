1. System Overview

FlowDesk is a web-based document approval and workflow management system. It manages the lifecycle of accounting and purchase documents through a "Review → Approve/Reject → Archive" pipeline. Documents enter the system via direct web upload or inbound webhooks (Google Forms / n8n), and upon approval, an outbound webhook triggers archiving in external systems like Paperless-ngx.

2. Architecture Diagram

The system uses a decoupled frontend/backend architecture. The React SPA communicates with the Express API server via REST. The API server coordinates between PostgreSQL (data), Cloudinary (files), and external systems (n8n, Google Forms) via webhooks.

Data Flow Summary

3. User Roles & Permissions

The system enforces two distinct roles with different permission levels. Role checks happen at both the API middleware layer (server-side) and the UI layer (conditional rendering).

CapabilityManagerAccountant
View all documentsYesYes
View dashboard with statsYes (Manager Dashboard)Yes (Standard Dashboard)
Upload new documentsNoYes
Approve documentsYesNo
Reject documents (with note)YesNo
Delete documentsYesNo
Replace & resubmit rejected documentsNoOwn documents only
View webhook logsYesYes
View audit trailYesYes

Role Enforcement

Server-Side (Middleware)

  • authMiddleware — validates JWT token on all protected routes
  • requireManager — blocks non-managers from approve/reject/delete
  • requireAccountant — blocks non-accountants from upload/resubmit

Client-Side (UI)

  • Upload button only shown to accountants
  • Approve/Reject buttons only shown to managers
  • Resubmit button only shown to the original uploader
  • Manager sees "Manager Dashboard" heading

Resubmission Permission Logic

The resubmit action has a compound check: the user must be an accountant AND the document must have status "needs_correction" AND the user must be the original uploader (uploadedBy.id === user.id). This prevents one accountant from modifying another's rejected document.

4. Document Workflow

This is the core of FlowDesk. Every document moves through a defined lifecycle from upload to archive, with full audit trail at every step.

4.1 Upload & Metadata

When an accountant uploads a document, they provide:

FieldTypeExample
TitleFree text"Q1 Office Supplies Payment"
Document TypeDropdownPayment Voucher, Journal Voucher, Purchase Order, Invoice, Receipt, Credit Note
Related PartyFree text"Staples Inc."
DateDate picker2026-03-09
FilePDF, PNG, JPGMax 10MB

The file is uploaded via multipart/form-data to Cloudinary. The Cloudinary URL and file metadata are stored in PostgreSQL. An audit event "Document uploaded" is created automatically.

What happens on Approve:

  • Document status changes to "approved"
  • Manager's name and timestamp stored as approved_by and approved_at
  • Audit event: "Approved by manager"
  • Outbound webhook fires to configured archive endpoint (n8n)
  • Audit event: "Outbound webhook triggered"
  • Webhook log entry created with direction, URL, status, and payload

What happens on Reject:

  • Document status changes to "needs_correction"
  • Rejection note stored on the document
  • Audit event: "Rejected by manager" with the rejection note
  • Rejection banner appears on the document detail page

4.3 Resubmission Flow

When a document is rejected, the original uploader (accountant) sees a "Replace & Resubmit" button. On click:

  • A file picker opens for the corrected document
  • The new file replaces the old one on Cloudinary
  • Document version number increments (v1 → v2)
  • Status resets to "pending_review"
  • Rejection note is cleared
  • Audit event: "Resubmitted (version N)" recorded
  • Document re-enters the approval queue for the manager

Version Tracking

Each resubmission increments the document's version field. The version number is displayed as a badge in the UI (e.g., "Version 2"). The full history of actions is preserved in the audit trail, so there's a complete record of every upload, rejection, and resubmission.

7. Webhook Integration

FlowDesk supports both inbound and outbound webhooks, enabling integration with Google Forms, n8n, and archive systems like Paperless-ngx.

7.1 Inbound Webhooks (Google Forms → FlowDesk)

External systems can submit documents to FlowDesk by POSTing to /api/webhooks/inbound:

POST /api/webhooks/inbound Content-Type: application/json { "title": "Field Purchase - Site Materials", "type": "purchase_order", "relatedParty": "BuildRight Supplies", "date": "2026-03-09", "source": "Google Forms" } Response: 200 { "id": "uuid", "message": "Document created from webhook" }

The document is created with status "pending_review" and source "webhook". A webhook icon appears in the UI to distinguish webhook-sourced documents from web uploads.

7.2 Outbound Webhooks (FlowDesk → n8n / Paperless-ngx)

When a manager approves a document, FlowDesk automatically sends an outbound POST to the configured archive endpoint:

POST https://n8n.example.com/webhook/archive Content-Type: application/json { "documentId": "a980b3ce-d901-4db9-92ab-081d52d7dab8", "title": "Information Security Assignment", "type": "invoice", "status": "approved", "archiveTo": "paperless-ngx" } The webhook URL is configured in the webhook_config table. The response status (200, 500, etc.) is logged in webhook_logs.

Webhook Monitoring

Every webhook (inbound and outbound) is logged in the webhook_logs table with direction, URL, HTTP method, response status, full payload, and timestamp. The Webhook Logs page in the UI provides a real-time view of all webhook activity.

8. File Handling & Preview

Upload Pipeline

Browser selects file
→
FormData POST to API
→
Multer + Cloudinary Storage
→
URL stored in PostgreSQL
  • Max file size: 10MB
  • Allowed formats: PDF, PNG, JPG, JPEG
  • Storage: Cloudinary flowdesk/ folder with resource_type: auto
  • URL format: https://res.cloudinary.com/{cloud}/image/upload/v{ver}/flowdesk/{id}.pdf

Document Preview

PDFs are previewed inline using Cloudinary's image transformation API. The first page of each PDF is converted to a high-resolution JPG on the fly:

Original: .../upload/v1773069114/flowdesk/abc123.pdf Preview: .../upload/pg_1,w_800/v1773069114/flowdesk/abc123.jpg The transformation pg_1 extracts page 1, w_800 sets width to 800px. Cloudinary generates and caches the JPG automatically. Images (PNG, JPG) are displayed directly without transformation.

A "Download" link provides access to the original file. The preview renders as an <img> tag, ensuring cross-browser compatibility without requiring a PDF viewer plugin.

9. Authentication & Security

JWT Authentication

User submits email + password
→
Server verifies with bcrypt
→
JWT token generated (24h expiry)
→
Token stored in localStorage

The JWT payload contains: { id, role, name, iat, exp }. Every API request includes the token in the Authorization: Bearer <token> header. On 401, the frontend auto-redirects to login.

Security Measures

  • Password hashing: bcrypt with 10 salt rounds (passwords never stored in plaintext)
  • Role-based middleware: requireManager and requireAccountant enforce role checks server-side
  • CORS: Configured to allow only the production frontend domain and localhost for development
  • File validation: Multer enforces 10MB limit and allowed file extensions
  • SQL injection prevention: All database queries use parameterized queries ($1, $2 placeholders)
  • Token expiry: JWT tokens expire after 24 hours, requiring re-login
  • Auto-redirect on 401: Expired or invalid tokens trigger automatic redirect to login page

Session Recovery

On page load, the app checks for an existing token in localStorage and validates it by calling GET /api/auth/me. If the token is expired or invalid, the user is silently redirected to login. This provides persistent sessions across browser refreshes without storing sensitive data.

Docker Compose Structure (Planned)

The production docker-compose.yml will include three services: app (Node.js Express server serving both API and React frontend), db (PostgreSQL 17), and nginx (reverse proxy with SSL). The frontend will be built and served statically by the Express server. A single docker compose up command will start everything.

Database Migration

The migration script (server/src/db/migrate.ts) automatically creates all tables with CREATE TABLE IF NOT EXISTS. It runs on server startup, making the database setup fully automatic. No manual SQL is needed.
